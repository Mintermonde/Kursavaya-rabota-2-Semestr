# Полный анализ проекта KursMVVM

> **Дата анализа:** 21.06.2025  
> **Проект:** Автоматизация обменного пункта валют (Avalonia + C# + MVVM)  
> **Целевая платформа:** .NET 8, Avalonia UI 11.2

---

## 1. Общая характеристика проекта

### Назначение
Десктопное приложение для автоматизации учета клиентов, валют, кассиров и сделок по обмену валюты. Поддерживает две роли: **Администратор** и **Кассир**.

### Технологический стек
| Компонент | Технология | Версия |
|-----------|------------|--------|
| GUI Framework | Avalonia UI | 11.2.0 |
| MVVM Framework | ReactiveUI | 11.2.0 (в составе Avalonia) |
| База данных | SQLite (Microsoft.Data.Sqlite) | 8.0.10 |
| Графики | LiveCharts2 (SkiaSharp) | 2.0.0-rc5 |
| Excel | EPPlus | 7.4.0 |
| CSV | CsvHelper | 33.0.1 |
| PDF | iTextSharp.LGPLv2.Core | 3.7.1 |
| Хеширование паролей | BCrypt.Net-Next | 4.0.3 |
| .NET SDK | .NET 8 | net8.0 |

### Функциональные возможности
- Авторизация с ролевой моделью (Admin / Cashier)
- Калькулятор обмена валют с автоматическим расчетом
- CRUD-операции для клиентов, кассиров, валют, сделок
- История сделок (кассир видит только свои)
- 5 типов отчетов (клиенты, валюты, кассиры, неактивные клиенты, прибыль)
- 4 типа графиков (LiveCharts2: столбчатые, круговые, линейные, гистограмма)
- Импорт/экспорт в JSON, XML, CSV, Excel
- Журнал аудита всех действий
- Интеграция с ЦБ РФ для обновления курсов
- Автосоздание БД с тестовыми данными

## 2. Архитектура проекта

### 2.1 Структура папок
```
KursMVVM/
├── Assets/                    # Ресурсы (не используются в коде)
├── Converters/                # 2 конвертера (Equals, DateTimeOffset)
├── Models/                    # 6 моделей данных
├── Services/                  # 2 сервиса (DataBaseService, AuthService)
├── ViewModels/                # 11 ViewModels
├── Views/                     # 11 Views (.axaml)
├── App.axaml / App.axaml.cs   # Точка входа, инициализация
├── Program.cs                 # Инициализация Avalonia
├── ViewLocator.cs             # Автоматическое сопоставление VM → View
├── init_database.sql          # SQL-скрипт для ручной инициализации
└── README.md                  # Документация
```

### 2.2 Архитектурный паттерн: MVVM
Проект следует паттерну **MVVM** с использованием **ReactiveUI**:
- **Models** — чистые модели с `INotifyPropertyChanged`
- **ViewModels** — наследуют `ReactiveObject` (через `ViewModelBase`), используют `RaiseAndSetIfChanged`
- **Views** — `.axaml` файлы с привязкой к ViewModels
- **ViewLocator** — автоматическое сопоставление по соглашению об именовании (`MyViewModel` → `MyView`)

### 2.3 Диаграмма зависимостей (упрощенная)
```
App.axaml.cs
  ├─ DataBaseService (SQLite)
  ├─ AuthService (зависит от DataBaseService)
  └─ MainWindowViewModel (зависит от обоих)
       └─ NavItem[] → ContentControl с DataTemplates
            ├─ CalculatorViewModel
            ├─ DealsViewModel
            ├─ ClientsViewModel
            ├─ CashiersViewModel
            ├─ CurrenciesViewModel
            ├─ ReportsViewModel
            ├─ ChartsViewModel
            ├─ ImportExportViewModel
            └─ AuditViewModel
```

## 4. Проблемы и недостатки

### 4.1 Критические архитектурные проблемы

#### 4.1.1 "Божественный класс" DataBaseService
- **Файл:** `Services/DataBaseService.cs` (~1100 строк)
- **Проблема:** Класс отвечает за:
  - Создание и миграцию схемы БД (12 методов)
  - CRUD для всех 5 сущностей (~20 методов)
  - Аутентификацию и управление пользователями
  - Аудит-логирование
  - 5 типов отчетов
  - 4 типа данных для графиков
  - Bulk-импорт
  - Интеграцию с ЦБ РФ
- **Нарушение:** Single Responsibility Principle (SRP)
- **Рекомендация:** Разделить на отдельные репозитории:
  ```
  IClientRepository, ICurrencyRepository, ICashierRepository, IDealRepository
  IAuditService, IReportService, IChartDataService, IImportExportService
  ```

#### 4.1.2 Нарушение MVVM: ViewModels создают UI-элементы
- **Места:**
  - `DealsViewModel.ShowConfirmDialogAsync()` — создает `Window`, `Button`, `StackPanel`
  - `ImportExportViewModel.ShowOpenFileDialogAsync()` / `ShowSaveFileDialogAsync()` — создает диалоги напрямую
- **Проблема:** ViewModel зависит от UI-фреймворка (Avalonia), невозможно unit-testировать
- **Рекомендация:** Внедрить абстракцию `IDialogService` / `IFileDialogService`

#### 4.1.3 Отсутствие Dependency Injection
- **Проблема:** Все зависимости создаются вручную в `App.axaml.cs` и передаются через конструкторы
- **Последствия:** Жесткая связанность, сложность тестирования, невозможность заменить реализацию
- **Рекомендация:** Внедрить `Microsoft.Extensions.DependencyInjection` или `Splat` (встроен в ReactiveUI)

#### 4.1.4 Отсутствие интерфейсов для сервисов
- **Проблема:** Все ViewModels зависят от конкретных классов (`DataBaseService`, `AuthService`)
- **Рекомендация:** Определить интерфейсы:
  ```csharp
  public interface IDataService { ... }
  public interface IAuthService { ... }
  ```

### 4.2 Проблемы с асинхронностью и потоками

#### 4.2.1 `async void` — антипаттерн
```csharp
// LoginViewModel.cs:99
private async void LoadCashiersAsync()  // ❌ async void — невозможно обработать исключения
```
- Исключения в `async void` крашат приложение
- Невозможно дождаться завершения

#### 4.2.2 Fire-and-forget без обработки ошибок
```csharp
// DealsViewModel.cs:57, CalculatorViewModel.cs:86
_ = LoadDealsAsync();       // ❌ Ошибки игнорируются
_ = LoadDataSafeAsync();     // ❌ Ошибки игнорируются
```

#### 4.2.3 Синхронный доступ к БД в асинхронном контексте
```csharp
// ВСЕ методы DataBaseService — синхронные
public List<Client> GetAllClients() { ... }  // ❌ Блокирует поток
```
- Рекомендация: Использовать `async/await` с `Microsoft.Data.Sqlite`:
  ```csharp
  public async Task<List<Client>> GetAllClientsAsync(CancellationToken ct = default)
  ```

#### 4.2.4 Некорректная работа с ObservableCollection из фоновых потоков
```csharp
// ClientsViewModel.cs:83, CashiersViewModel.cs:72, и др.
Clients = new ObservableCollection<Client>(clients);  // ⚠️ Иногда работает в Avalonia, но не гарантировано
```
- Создание новой коллекции в `Task.Run` может вызывать race conditions
- Рекомендация: Всегда использовать `Dispatcher.UIThread.Post()` для модификации коллекций

### 4.3 Проблемы безопасности

#### 4.3.1 Пустые catch-блоки
```csharp
// DataBaseService.cs:373
catch { /* Аудит не должен ломать основную логику */ }  // ❌ Глотает ВСЕ исключения
```
- Даже `OutOfMemoryException` или `StackOverflowException` будут проигнорированы
- Рекомендация: Логировать минимум `catch (Exception ex) { Log(ex); }`

#### 4.3.2 Демо-пароли в README
```markdown
| admin | admin | Администратор |
| cashier1 | cashier1 | Кассир #1 |
```
- Не критично для курсовой, но в production недопустимо

#### 4.3.3 Нет защиты от SQL-инъекций в некоторых местах
- Хотя в основном используются параметризованные запросы, в `ReadClient`, `ReadCurrency` и др. нет проверки на `DBNull`/`null` для обязательных полей

### 4.4 Проблемы с памятью и утечками

#### 4.4.1 Неотписанные события
```csharp
// App.axaml.cs:60, 81, 93
loginVm.LoginSuccess += (s, e) => { ... };
mainVm.RequestLogout += (s, e) => { ... };
```
- `MainWindow` и `LoginWindow` не закрываются, а скрываются (`Hide()`)
- События никогда не отписываются → утечка памяти при многократных логинах/логаутах

#### 4.4.2 HttpClient не использует HttpClientFactory
```csharp
// DataBaseService.cs:520
using var client = new HttpClient();  // ⚠️ Создается на каждый вызов
```
- Для редких вызовов допустимо, но `HttpClient` лучше кэшировать

### 4.5 Проблемы с кодом и стилем

#### 4.5.1 Смешанные окончания строк (CRLF / CR)
- `Program.cs` и `AuthService.cs` содержат одиночные `\r` (CR) — признак некорректной работы Git на Windows

#### 4.5.2 Магические строки и числа
```csharp
// Повторяющиеся магические строки:
"RUB"        // >20 раз в коде
"#2980B9"    // Цвета в XAML и ViewModel
"yyyy-MM-dd" // Формат даты в 10+ местах
"dd.MM.yyyy" // Другой формат даты
```

#### 4.5.3 "Мертвый" код
```csharp
// CurrenciesViewModel.cs — неиспользуемый using
using System.Drawing;  // ❌ Не используется
```

#### 4.5.4 Debug-вывод в production
```csharp
// MainWindowViewModel.cs — множество Debug.WriteLine
System.Diagnostics.Debug.WriteLine($"=== SelectedNavItem SETTER...");
```

#### 4.5.5 Использование русских идентификаторов в SQL
```sql
CREATE TABLE Клиенты (номер_клиента INTEGER, фио TEXT);
```
- Потенциальные проблемы с кодировками на разных системах
- Сложность поддержки для не-русскоязычных разработчиков

### 4.6 Проблемы с тестируемостью

| Проблема | Влияние |
|----------|---------|
| Нет интерфейсов | Невозможно мокировать зависимости |
| ViewModel создает UI | Невозможно тестировать диалоги |
| Синхронный доступ к БД | Тесты будут медленными и нестабильными |
| `App.axaml.cs` создает все вручную | Невозможно подменить сервисы для тестов |
| Нет отдельного слоя бизнес-логики | Вся логика в DataBaseService (инфраструктура) |

### 4.7 Недостающая функциональность

| Функция | Статус | Комментарий |
|---------|--------|-------------|
| Unit-тесты | ❌ Отсутствуют | Нет ни одного тестового проекта |
| Валидация данных | ⚠️ Частичная | Нет валидации паспортного номера, email, телефона |
| Отмена операций | ❌ Нет | Нет CancellationToken |
| Локализация | ❌ Нет | Только русский язык, жестко закодирован |
| Печать отчетов | ❌ Нет | PDF-экспорт подключен, но не используется в UI |
| Поиск/фильтрация | ❌ Нет | Нет поиска по клиентам, сделкам |
| Пагинация | ❌ Нет | Все данные загружаются сразу |
| Резервное копирование БД | ❌ Нет | Нет функции backup |
| Логирование ошибок | ⚠️ Частичное | Только `Debug.WriteLine` |

## 6. Рекомендации по улучшению

### 6.1 Немедленные (критические)
1. **Разделить DataBaseService** на репозитории и сервисы
2. **Убрать `async void`** — заменить на `async Task` + корректный вызов
3. **Добавить обработку ошибок** в fire-and-forget вызовах
4. **Убрать UI-зависимости** из ViewModels (диалоги, окна)
5. **Добавить интерфейсы** для всех сервисов

### 6.2 Краткосрочные (важные)
6. **Внедрить Dependency Injection** (Splat или MS.DI)
7. **Перевести DataBaseService на async/await**
8. **Добавить CancellationToken** в длительные операции
9. **Исправить утечки памяти** — отписываться от событий, закрывать окна
10. **Вынести магические строки** в константы/конфигурацию
11. **Добавить поиск и фильтрацию** в DataGrid

### 6.3 Долгосрочные (желательные)
12. **Добавить unit-тесты** (xUnit + Moq + FluentAssertions)
13. **Внедрить бизнес-логику** отдельно от доступа к данным
14. **Добавить пагинацию** для больших таблиц
15. **Реализовать логирование** (Serilog или NLog)
16. **Добавить резервное копирование** БД
17. **Перевести SQL-идентификаторы** на английский (или использовать ORM)
18. **Добавить валидацию** моделей (FluentValidation)
